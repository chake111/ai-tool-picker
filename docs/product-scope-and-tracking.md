# 产品边界、埋点与评审冻结（V1）

> 适用范围：当前迭代（首页 / 结果页 / 对比流程）

## 1. 产品边界与不做项

### 1.1 首页（Home）
**目标边界**
- 只承载「问题输入 + 快速场景入口 + 搜索触发」，帮助用户在最短路径进入推荐结果。
- 首页不承担深度筛选、复杂配置和对比决策，仅做意图收集与首跳转化。

**不做项（Out of scope）**
- 不在首页展示完整工具长列表（避免信息过载）。
- 不在首页承载多维度高级筛选（价格、平台、语言等放在结果页）。
- 不在首页做工具详情展开（详情仅在结果卡片与外链完成）。

### 1.2 结果页（Results）
**目标边界**
- 承载推荐列表、基础筛选、收藏、加入对比、访问官网、Refine（二次收敛）等核心决策动作。
- 结果页对用户给出的 query 负责，确保「可解释推荐 + 可继续缩小范围」。

**不做项（Out of scope）**
- 不做跨 query 的自动合并推荐（每次结果由当前 query 驱动）。
- 不做复杂 BI 分析视图（运营看板在外部数据平台）。
- 不做站内完整评测内容生产（仅提供推荐理由与跳转）。

### 1.3 对比流程（Compare）
**目标边界**
- 对比流程仅服务「候选集收敛」：支持从结果页加入候选、在对比页并排查看后做最终选择。
- 对比页强调横向差异（价格、平台、语言、适配场景），不重复推荐逻辑。

**不做项（Out of scope）**
- 不做无限制批量对比（保持可读性，采用上限策略）。
- 不做对比结果自动排序重算（避免用户心智混乱，排序以当前展示规则为准）。
- 不做对比页内再次召回新工具（新增候选回到结果页完成）。

## 2. 事件埋点清单（本轮范围）

> 目标：覆盖从检索、筛选、收敛到外跳的关键行为，统一事件命名与字段口径。

| 事件名称 | 触发时机 | 必填字段 | 建议扩展字段 | 对应阶段 |
| --- | --- | --- | --- | --- |
| `search_submit` | 用户提交搜索（按钮/回车） | `query`, `locale`, `request_id`, `timestamp` | `entry`(hero/quick_scene), `ranker` | 发现 |
| `filter_apply` | 用户切换筛选条件 | `filter_key`, `filter_value`, `query`, `request_id` | `result_count_before`, `result_count_after` | 收敛 |
| `compare_add` | 用户将工具加入对比 | `tool_id`, `query`, `request_id` | `position`, `compare_size` | 收敛 |
| `favorite_add` | 用户收藏工具 | `tool_id`, `query`, `request_id` | `position`, `from`(result/compare) | 偏好沉淀 |
| `visit_official_site` | 用户点击官网外链 | `tool_id`, `query`, `request_id` | `position`, `target_domain` | 转化 |
| `refine_submit` | 用户在结果页触发 refine（二次输入） | `query`, `refine_query`, `request_id` | `refine_count`, `delta_tokens` | 收敛 |

### 2.1 与当前实现映射（便于研发落地）
- 现有基础事件：`search`、`click`、`favorite`、`exposure`、`impression`（见 `ui/lib/track.ts`）。
- 建议映射策略：
  - `search_submit` → `search` + `metadata.entry`
  - `compare_add` / `visit_official_site` / `refine_submit` / `filter_apply` → `click` 或 `search` + 强约束 `metadata.action`
  - `favorite_add` → `favorite` + `operation=add`
- 若需要 BI 直连，也可在服务端保留统一抽象事件，同时输出兼容字段到现有事件模型。

## 3. 分阶段验收口径（视觉 / 交互 / 数据）

### 阶段 A：首页（输入与触发）
- **视觉验收**
  - 首屏包含主标题、输入框、主 CTA，且移动/桌面两端不出现布局错位。
  - 快捷场景入口样式统一，hover/active/disabled 状态清晰。
- **交互验收**
  - 空输入提交有明确错误提示；有效输入可成功触发推荐请求。
  - 回车与按钮点击行为一致，且重复提交期间有 loading 态。
- **数据验收**
  - 每次有效提交产生 1 条 `search_submit`。
  - `query`、`locale`、`request_id` 字段完整率 100%。

### 阶段 B：结果页（筛选与二次收敛）
- **视觉验收**
  - 推荐卡片信息层级一致（名称、理由、标签、关键属性）。
  - 筛选控件在不同断点下可用且无遮挡。
- **交互验收**
  - 筛选切换后列表即时更新，分页与结果数同步变化。
  - Refine 输入可触发新一轮推荐并保留必要上下文（如 query 来源）。
- **数据验收**
  - 每次筛选触发 1 条 `filter_apply`，before/after 结果数可回放。
  - 每次 refine 触发 1 条 `refine_submit`，并可关联原始 `request_id`。

### 阶段 C：对比与决策（Compare / 收藏 / 外跳）
- **视觉验收**
  - 对比表头与行内容对齐，关键维度（价格、平台、语言）可直接横向阅读。
  - 收藏态、已加入对比态在卡片与对比页展示一致。
- **交互验收**
  - 从结果页加入对比后，对比页可即时看到目标工具。
  - 收藏状态可切换并在刷新后保持一致；官网按钮可正常外跳。
- **数据验收**
  - `compare_add`、`favorite_add`、`visit_official_site` 事件触发准确率 100%。
  - 事件中的 `tool_id`、`request_id`、`position` 可用于还原用户路径。

## 4. 30 分钟评审会组织方案（需求冻结）

**会议目标**
- 在 30 分钟内确认：边界、不做项、埋点口径、验收标准。
- 会后进入执行态，默认不再新增临时需求（除 P0 线上事故）。

**参会角色（建议）**
- PM（主持与决策）
- 设计（视觉与交互验收代表）
- 前端/后端（实现可行性与排期承诺）
- 数据分析（埋点字段与看板口径）
- QA（验收与回归策略）

**时间切片（30 min）**
1. 0-5 min：复述目标与范围（含不做项）
2. 5-12 min：确认首页 / 结果页 / 对比流程边界
3. 12-20 min：逐条确认 6 个核心埋点字段口径
4. 20-27 min：确认阶段验收标准（视觉/交互/数据）
5. 27-30 min：冻结结论、责任人与时间点

**会议输出（必须落文档）**
- 冻结版需求清单（包含不做项）。
- 冻结版埋点字典（事件名、字段、示例值）。
- 验收 checklist 与 owner。
- 变更控制规则：会后新增需求默认进入下一迭代。

**临时需求控制机制（建议）**
- 本迭代仅接受两类变更：
  1) P0 线上故障修复；
  2) 法务/合规强制要求。
- 其他新增需求统一登记 backlog，由 PM 在下个迭代评审。
